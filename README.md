# uni-go
基于 Agent 的 UniGo 智能旅游助手：长链推理与多工具协同实践
本项目基于 Spring AI 框架构建了面向大学生的一站式旅游智能体系统——UniGo。核心成果并非简单对话机器人，而是一个具备感知、推理、执行与记忆能力的 Agent，能够将“周末去成都，预算五百块”这类模糊需求自动转化为包含路线、景点、预算分配及实时调整的完整行程。

一、解决的核心痛点
大学生旅游面临三大难题：一是信息过载，需在多平台手动整合景点、交通、票价等碎片信息，耗时数小时且易遗漏关键约束；二是预算敏感但缺乏动态反馈，现有推荐很少实时计算总花费是否超支；三是行程调整困难，天气变化或景点关闭后需手动重排，极易产生冲突与绕路。这些问题本质上是“多源检索+多约束优化”的复合决策任务，传统 APP 和通用大模型单轮问答均无法胜任。

二、核心逻辑流：长链推理 + 单 Agent 多工具编排
UniGo 未采用多 Agent 协作，而是基于 ReAct 模式设计了一个单 Agent 多工具闭环系统，更贴合旅游规划串行推理、逐步执行、动态修正的特征。

1. 意图澄清与任务分解（长链推理起点）
用户输入自然语言后，Agent 首先进行场景化解析。例如“清明三天重庆穷游”会被拆解为时间窗口、目的地、预算（单日≤150元）、密度偏好等结构化子任务。关键信息缺失时，Agent 主动发起多轮对话引导补充。本阶段不调用外部工具，完全依赖 Prompt 中的推理链路输出任务清单。

2. 工具调用与信息获取
根据子任务，Agent 自动编排工具序列：先调用高德地图获取 POI 列表，并行请求天气 API，再针对候选景点查询开放时间、票价、评分。编排顺序由 Agent 动态决定——如预报大雨则优先推荐室内景点。

3. 多约束融合与行程生成（核心推理）
Agent 将景点信息与预算、时间、地理可达性联合优化。采用约束满足与启发式排序：过滤超预算组合，按“评分/价格比”贪心填充时间槽，保证相邻景点交通耗时≤45分钟。生成后做一致性校验，避免闭馆、绕路等逻辑幻觉。

4. 实时调整与记忆更新
用户可随时变更指令（如“第二天不去磁器口，换涂鸦街”）。Agent 识别受影响时段，锁定已执行部分，仅对剩余窗口重新推理与调用工具，更新短期记忆（Redis）。这种局部重规划显著降低延迟，避免前后矛盾。

5. RAG 与社交作为辅助决策
Agent 在推理中动态检索旅游垂直知识库（RAG），例如从攻略中召回“洪崖洞避开人群”的真实经验。社交匹配模块独立运行，基于画像推荐旅伴，与主流程松耦合。

三、实践成果
目前 UniGo 已完成端到端联调。在 50 组盲测中，工具调用成功率 92%，预算符合率 88%，平均响应时延 4.2 秒。相比直接调用通用大模型，方案在“可执行性”上提升约 40 个百分点，动态调整平均仅需 1.8 秒。项目验证了 Spring AI 在 Java 生态下构建长链推理 Agent 的可行性，形成了可复用的架构模板。
